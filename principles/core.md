# Базовые принципы Block Architecture

## Принцип 1: Изоляция по результату

Блок знает только:
- Что он получает на вход
- Что должен отдать на выход

Внутренняя реализация — чёрный ящик. Никто не лезет внутрь пока результат соответствует контракту.

## Принцип 2: Контракт важнее кода

Если результат соответствует контракту — блок работает. Не важно как реализовано внутри.

Это освобождает от code review в традиционном смысле. Ревьюится контракт, не код.

## Принцип 3: Диагностика по цепочке

```
Вход → [Блок 1] → Результат ✓ → [Блок 2] → Результат ✓ → [Блок 3] → Результат ✗
```

Не нужно читать код. Нужно найти где результат стал красным.

Если блок получает несколько входов — смотрим какой вход красный:

```
Блок А → ✓ ─┐
Блок Б → ✓ ─┼→ [Блок Д] → ?
Блок В → ✗ ─┘

Проблема очевидна: Блок В
```

## Принцип 4: Один блок — один агент

Каждый блок может обслуживаться отдельным AI-агентом который:
- Не знает про другие блоки
- Знает только свой контракт
- Отвечает за свой результат

## Принцип 5: Человек — архитектор

Человек определяет:
- Какие блоки существуют
- Какие контракты между ними
- Решает тупиковые ситуации

Человек НЕ:
- Пишет код
- Читает код
- Ревьюит код

## Аналогия с IT-командой

| Традиционная команда | Block Architecture |
|---------------------|-------------------|
| PM видит процесс | Человек видит поток результатов |
| Backend не знает код фронта | Агент не знает код других блоков |
| API контракты между сервисами | Контракты между блоками |
| Микросервисы | Блоки |

Разница: вместо людей на каждом блоке — AI-агенты.
